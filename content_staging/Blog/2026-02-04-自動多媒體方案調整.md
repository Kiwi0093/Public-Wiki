# 前言

為了改善困擾我很久的`lidarr` API連接`Musicbrainz`的問題,我針對我的<ruby><font color="lightblue"><del>自動白嫖盜版影音方案</del></font><rp>(</rp><rt><font color="red">自動多媒體收集方案</font></rt><rp>)</rp></ruby>做了一些調整

<!--truncate-->

# 調整內容

## Lidarr

### 事故

起因是因為`Musicbrainz`變更了API相關設定,導致`lidarr`無法正常的從`Musicbrainz`取得Meta資料,所以`lidarr`的功能基本上就廢了

### 對策

其實簡單的來說,最終的解決方案就是 - <ruby><font color="lightgreen" size="+3">等待</font></ruby>, 實際上等的夠久後`lidarr`的開發團隊就提供了改版的`lidarr`去搭配`Musicbrainz`的新設定(但是text search的部份還是沒救)
<ruby><font color="red" size="+3">BUT</font></ruby>因為大家之前都等不及,所以在開發團隊解決之前,紛紛用上了其他改過得版本,或是加上mod.結果就是用了改過版本的直接改新版也是無效的,所以才有後面悲苦的相關經歷

### 曲折的過程

因為大叔我<ruby><font color="lightblue"><del>主動積極導入當時的最佳解決方案</del></font><rp>(</rp><rt><font color="red">等不了六七個月,一時手賤</font></rt><rp>)</rp></ruby>,所以改版後只能把整個`lidarr`的config<ruby><font color="red" size="+3">全部砍掉</font></ruby>,然後再重新設定與導入檔案....恩..過程還真的是漫長且折騰....



### 新舊差異

最大的差異就是,改好的`lidarr`依舊沒辦法利用`text search`跟之前一樣直接搜尋歌手名字或是專輯名稱就可以`add new`,而是需要先去`Musicbrainz`查詢該歌手的`Artist ID`或是該專輯的`Release group ID`

然後到`lidarr`的`add new`裡面使用

```bash
lidarr:<Artist ID/Release group ID>
```

然後如同以往,選擇跑出來的圖面上,加入到`lidarr`內

## 網路相關

### Automedia方案調整

原先因為參考資料,所以我把整個automedia方案(包括`radarr`,`sonarr`, `lidarr`, `jackett`...)都包在`Gluten`裡面,但是因為最近`Gluten`的VPN有點問題(其實我也不確定有問題多久了)
所以先是發現`jackett`+`flaresolverr`怪怪的,所以將他們這對搭當移出`Gluten`
接著是`lidarr`修正中(對就是上面講的...)發現`add new`的search連不上...果斷將`lidarr`移出`Gluten`後就好了...
於是就是把整個automedia方案除了`qbittorrent`還留在`Gluten`內以外其他都移出,畢竟這樣才是我理想的只有`qbittorrent`走VPN保平安,其他東西就不要佔用VPN的頻寬了....
然後我才發現,怎麼參考資料裡說的不放在一起就會連不到的問題不見了...(可能是因為我有刻意把automedia內的每個container都指定了ip的關係...)

### VPN - ERROR [vpn] getting public IP address information:

在`docker-compose.yml`的`enviroment`內加上`PUBLICIP_ENABLED=false`就好了
